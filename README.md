1)  C Shell :
I implements a custom C Shell in C, designed a wide range of shell functionalities. The shell displays a prompt in the format <Username@SystemName:current_path> where the home directory is abbreviated as ~, accepts user input, and parses commands according to a specified context-free grammar, printing “Invalid Syntax!” for invalid commands. It supports built-in intrinsics such as hop for changing directories, reveal for listing files with optional flags -a and -l, and log for maintaining a persistent history of the last 15 commands with options to display, purge, or execute a previous command. The shell also allows execution of arbitrary commands, input and output redirection using <, >, and >>, and piping using |. Sequential execution of commands using ; and background execution with & are fully supported. Advanced job control features include activities to list running or stopped processes, ping to send signals to processes, and handling of keyboard shortcuts such as Ctrl-C to send SIGINT to foreground processes, Ctrl-D to terminate the shell and kill all child processes, and Ctrl-Z to stop foreground processes and move them to the background. The shell also supports fg and bg commands to bring background or stopped jobs to the foreground or resume them in the background. The project is implemented in C using only POSIX-compliant functions and libraries, compiled with GCC using strict POSIX flags to ensure standard compliance. The shell can be compiled with make all and executed with ./shell.out, providing a fully functional and interactive environment for command execution, job control, and process management

2) NETWORKING :

In networking i implements a reliable data transfer protocol on top of UDP, similar to a simplified TCP, which I called the SHAM protocol. Both a client and a server were written to communicate using custom headers that include sequence numbers, acknowledgment numbers, flags (SYN, ACK, FIN), and a window size field. The system supports two modes of operation: file transfer and chat. In file transfer mode, the client sends a file to the server, which writes it locally and verifies its integrity using an MD5 checksum. In chat mode, both client and server can exchange messages interactively until either side sends the /quit command. To ensure reliability, the protocol uses a three-way handshake for connection establishment (SYN, SYN-ACK, ACK), stop-and-wait retransmission with timeouts, acknowledgments for each data segment, and a four-way FIN handshake for clean connection termination. Packet loss can be simulated by specifying a probability value, allowing the protocol to be tested under unreliable network conditions. Each packet transmission, acknowledgment, retransmission, and timeout event is logged with timestamps in client_log.txt and server_log.txt for debugging and analysis. The system can be built with gcc (the server additionally links OpenSSL for MD5 hashing) and run by starting the server with a port number and optional --chat flag, then connecting a client with the server’s IP, port, file arguments or chat mode, and a packet loss probability. Overall, this project demonstrates how to build reliability, ordering, and session management on top of UDP by re-creating key features of TCP, including handshakes, retransmissions, acknowledgments, and graceful termination.



3) xv6 :

In this project, we enhanced the xv6 operating system by making readcount tracks and returns the number of bytes read by the read() system call throughout all processes since boot and by implementing two new scheduling algorithms: First Come First Serve (FCFS) and a simple Completely Fair Scheduler (CFS), alongside the standard round robin. The scheduler may be specified at compile time using the SCHEDULER macro (e.g., make qemu SCHEDULER=FCFS or make qemu SCHEDULER=CFS).

For FCFS, we added a field ctime to every process (in struct proc) to save creation time. The scheduler always executes the runnable process with the minimum ctime, i.e., the process that arrived first. Processes execute until completion or blocking, and there is no preemption.

For CFS, we added nice, weight, and vruntime fields in struct proc. Every process is assigned a weight depending on its nice value, and vruntime measures how much CPU time it's received so far (normalized by weight). The scheduler always selects the runnable process with the minimum vruntime. The time slice is calculated as target_latency / number_of_runnable_processes (with target latency = 48 ticks, min slice = 3). Upon running, a process's vruntime is then refreshed by p->vruntime += (ticks_ran * 1024) / p->weight.

We also modified allocproc() to set these new fields and included debug prints in the scheduler to indicate which process was being scheduled. Testing was performed utilizing existing programs such as readcount and an optional user program schedulertest to check for fairness.
